What is Apex Runner?

Apex Runner is a lightweight browser-based tool for running Apex scripts stored in a GitHub repository against any Salesforce sandbox. It lives as a single index.html file on GitHub Pages — no server, no backend, no installation required.

It was built to solve a common problem: when a new sandbox is created, a set of post-deployment Apex scripts needs to be executed to set up reference data (approval rules, CPQ configuration, etc.). Doing this manually — opening Developer Console, pasting each script one by one — is slow and error-prone.

No infrastructure needed. The app is a static HTML file. It talks directly to Salesforce and GitHub APIs from your browser. Nothing is stored on any server.

How it works

Step 1 — Connect to Salesforce

Click Login with Salesforce SSO. You'll be redirected to your org's standard Salesforce login page — the same one you use every day, with your SSO button. After authenticating, you're returned to the app with a session token. The token lives only in sessionStorage and is gone when you close the tab.

Step 2 — Select scripts

Choose where to load scripts from — a ZIP archive downloaded from GitHub, or directly from a GitHub repository. Select the folders you want to run. Scripts execute in alphabetical order within each folder, so naming them 01_setup.txt, 02_configure.txt controls execution order.

Step 3 — Run

The app sends each script to Salesforce via the Tooling API (executeAnonymous endpoint) over HTTPS. Results appear in real time — each script shows pass ✅ or fail ❌ with the error message and execution time. When finished, you can export a full HTML report.

Deploying to GitHub Pages

  1. Create a new GitHub repository (can be public or private if your org is on Team/Enterprise plan)
  2. Place index.html in the root of the repo
  3. Fill in the CONFIG block at the top of the script section
  4. Go to Settings → Pages → Source: Deploy from branch → main / root
  5. After 1–2 minutes the app is live at https://your-org.github.io/repo-name/
Public repo + CONFIG token: If your Pages repo is public, do not hardcode the GitHub token in CONFIG.GH_TOKEN — anyone can read it from the source. Instead, leave it empty and each user enters their own PAT in the GitHub tab on step 2. It gets saved to their browser's localStorage.

Salesforce login (SSO)

The app uses OAuth 2.0 PKCE flow with the public PlatformCLI client ID — the same one Salesforce CLI uses internally. This means no Connected App setup is required.

PKCE (Proof Key for Code Exchange) is a secure OAuth flow designed for browser-based apps. It does not require a client secret. When you click Login, the app generates a one-time code challenge, redirects you to Salesforce, and exchanges the returned code for an access token — all without any server involvement.

SSO works transparently: Salesforce's login page shows your company's SSO button exactly as it does when you log in normally. The app doesn't need to know anything about your SSO provider.

If you want to use your own Connected App instead of PlatformCLI (for example to restrict access to specific users), open Advanced settings on the Connect screen and enter your Consumer Key. Setup: App Manager → New Connected App → Enable OAuth → PKCE enabled, no secret required, callback URL set to your GitHub Pages URL.

Script sources

📦 ZIP archive

Download the repo as a ZIP from GitHub (Code → Download ZIP) and drag it into the app. No GitHub token needed. The app reads the ZIP entirely in browser memory — nothing is uploaded anywhere. Use this when you don't want to deal with tokens, or when working offline.

🔗 GitHub repository

Connect directly to a private GitHub repo using a Personal Access Token. The token is saved in your browser's localStorage so you only enter it once per device. The app uses the GitHub Contents API and Git Trees API to browse the repo structure and fetch file contents over HTTPS.

Repo structure

Scripts should be organized in subfolders under a base path. The app displays each subfolder as a collapsible group with checkboxes. Both .txt and .apex extensions are supported by default.

scripts/
  apex/
    Post Deployment Scripts/     ← set this as base path
      Approval Rules/
        Credit Routers/
          CPQ1-168106.txt
          CPQ1-183625.txt

      CPQ Setup/
        01_price_rules.txt
        02_discount_schedules.txt

Scripts within each folder run in alphabetical order. Prefix filenames with numbers (01_, 02_) to control execution order explicitly.

There is no backend server. No data is logged, tracked, or stored anywhere except your own browser. All API calls go directly from your browser to Salesforce or GitHub over HTTPS.

GitHub token permissions

Use a fine-grained PAT with the minimum permissions:

  • Repository access: Only select repositories → choose the scripts repo only
  • Repository permissions → Contents: Read-only
  • Everything else: No access

If your org enforces SAML SSO on GitHub, after creating the token click Configure SSO → Authorize next to your organization name.

For a shared token used by the whole team, create it on a bot/service account rather than a personal account — so the token doesn't break if someone leaves the company.

Config reference

Edit the CONFIG block near the top of the <script> section in index.html:

const CONFIG = {
  // GitHub read-only PAT for fetching scripts from the repo.
  // Leave empty if users will enter their own token in the UI.
  // WARNING: do not set this if index.html is in a public repo.
  GH_TOKEN: '',            // e.g. 'ghp_xxxxxxxxxxxx'

  // Default repo shown on step 2 (GitHub mode).
  DEFAULT_REPO: '',        // e.g. 'myorg/salesforce-scripts'

  // Base path inside the repo where script folders live.
  DEFAULT_PATH: 'scripts/apex/Post Deployment Scripts',

  // File extensions to look for (comma-separated).
  DEFAULT_EXTS: '.txt, .apex',
};

FAQ

Do I need a Salesforce Connected App?

No. The app uses the public PlatformCLI client ID which works on any org without any setup. If you want tighter control over who can use the app, you can create your own Connected App and enter its Consumer Key in Advanced settings.

Can I run scripts on production?

Technically yes — the app works against any org you can log into. But it's designed for sandboxes. Be careful. The HTML report gives you a full audit trail of what ran and what the outcome was.

What happens if a script fails?

The app logs the error message returned by Salesforce (compile error, exception message, etc.), marks the script as ❌ failed, and continues with the next script. It never stops mid-run. After finishing, the summary shows total passed / failed count, and you can export the full report.

How do I control the order scripts run in?

Scripts within a folder run alphabetically. Prefix filenames with numbers: 01_create_groups.txt, 02_assign_approvers.txt. Folders themselves also run in alphabetical order when all are selected.

The Salesforce login redirects me but nothing happens

Make sure the callback URL registered in your Connected App (or the default PlatformCLI app) matches exactly the URL of this page including the trailing slash. Also check that pop-ups are not blocked.

I get a 403 error when loading scripts from GitHub

Your PAT probably hasn't been authorized for your organization's SSO. Go to github.com → Settings → Personal access tokens, find your token, and click Configure SSO → Authorize next to your org name.
